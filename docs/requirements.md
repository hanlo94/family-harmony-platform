# 产品需求文档 (PRD)

## 家庭共享任务与"无唠叨"协作平台

**版本**：v1.1 MVP
**日期**：2026-06-11
**状态**：需求优化稿
**平台范围**：v1.0 仅做微信公众号 H5，代码结构与接口设计兼容后续微信小程序，不做原生 App

> 注意：本 PRD v1.1 调整了 MVP 范围和平台边界，`architecture.md`、`ui-design.md`、`database.md`、`api-design.md` 需要后续同步更新。

## 1. 产品定位与目标

### 1.1 产品愿景

> 让家庭家务分配更公平、协作更愉快，用系统提醒代替人际催促，减少家庭中的"精神负担"。

**一句话**：一个帮助家庭快速分配家务、自动提醒、轻量看见贡献的家庭协作 H5 应用。

### 1.2 MVP 目标

v1.0 的目标不是做完整家庭管理平台，而是完成一个低摩擦的家务协作闭环：

```
创建家庭 → 邀请成员 → 快速创建任务 → 分配执行人 → 自动提醒 → 成员完成 → 可选验收 → 家庭看板反馈
```

MVP 应重点解决：

- 家庭组织者不想反复安排和催促。
- 家庭成员不想靠口头提醒才知道自己要做什么。
- 周期性家务不应该每天重复手动创建。
- 关键任务可以验收，普通任务不应被过度管理。
- 微信提醒不可用时，首页仍能作为任务提醒兜底。

### 1.3 成功指标

| 指标 | 目标 |
|------|------|
| 首次建家完成率 | 用户登录后可顺利创建或加入家庭 |
| 首个任务创建完成率 | 家庭创建后可在 30 秒内创建首个任务 |
| 任务按时完成率 | 用户能通过提醒和首页待办及时处理任务 |
| 组织者手动催促次数 | 系统提醒和看板应减少人工催促 |
| 周活跃家庭数 | 家庭成员每周至少完成或查看任务 |

## 3. 目标用户与核心场景

### 3.1 目标用户

| 角色 | 核心诉求 | 典型场景 | 需要关注的风险 |
|------|----------|----------|----------------|
| **家庭组织者** | 减少安排、提醒、催促带来的精神负担 | 创建家庭、邀请成员、创建常用家务、查看整体进度 | 如果任务创建太麻烦，组织者仍然承担主要负担 |
| **成人家庭成员** | 清楚知道自己该做什么，并减少被催促的不适感 | 查看今日任务、收到提醒、完成任务、补充说明 | 如果提醒像"监工"，容易产生抵触 |
| **孩子/青少年成员** | 通过清晰任务培养责任感，获得正向反馈 | 查看自己的任务、完成后提交、等待家长确认 | 如果验收过重，容易变成惩罚或考核 |
| **多家庭用户** | 在多个家庭空间之间切换 | 自己家庭、父母家庭、合租空间等 | 任务和提醒必须明确所属家庭，避免混淆 |

### 3.2 核心场景

1. **首次建家**：用户微信登录后创建家庭，生成邀请链接或二维码，邀请伴侣、孩子或其他成员加入。
2. **快速创建家务**：组织者从模板选择"倒垃圾、洗碗、拖地"等任务，设置执行人、截止时间、重复规则和验收方式。
3. **成员执行任务**：成员打开首页查看今天要做什么，完成后可直接标记完成，也可补充备注或照片。
4. **无唠叨提醒**：系统在任务到期前提醒执行人；若微信提醒不可用，首页突出展示临近到期和逾期任务。
5. **验收与驳回**：开启验收的任务由组织者确认或驳回；驳回必须填写原因。
6. **轻量公平感**：家庭看板展示每个成员当前待办数和本周完成数，让贡献被看见，但不做排名。

### 3.3 产品原则

系统应该帮助家庭成员形成清晰协作，而不是制造监督感。默认设计应避免排名、惩罚、公开羞辱和过度验收。

## 5. 核心流程

### 5.1 家庭启动流程

```
微信登录 → 创建/加入家庭 → 邀请成员 → 选择或创建任务模板 → 创建首批任务 → 进入家庭任务看板
```

### 5.2 任务创建流程

```
选择模板/手动创建 → 填写标题、截止时间、执行人 → 选择重复规则 → 选择是否需要验收 → 创建成功
```

MVP 表单字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 标题 | 是 | 可由模板预填 |
| 描述 | 否 | 任务要求或注意事项 |
| 执行人 | 是 | 家庭成员之一 |
| 截止时间 | 是 | 用于提醒和逾期判断 |
| 难度 | 否 | 默认 1-5 星，用于轻量表达工作量 |
| 重复规则 | 是 | 默认不重复，支持每天、每周 |
| 是否需要验收 | 是 | 默认无需验收 |
| 提醒时间 | 否 | MVP 默认到期前 1 小时 |

### 5.3 完成任务流程

无需验收：

```
成员点击完成 → 填写可选备注/照片 → 任务变为已完成 → 首页反馈完成状态
```

需要验收：

```
成员点击完成 → 填写可选备注/照片 → 任务变为待验收 → 组织者确认/驳回
```

### 5.4 验收与驳回流程

```
组织者查看待验收任务 → 验收通过/驳回 → 驳回时填写原因 → 执行人重新处理
```

驳回必须填写原因，避免只产生负面反馈而没有改进信息。

### 5.5 重复任务生成流程

```
重复任务完成 → 系统生成下一次任务 → 继承标题、执行人、验收设置、提醒设置
```

MVP 只支持：

- 不重复
- 每天
- 每周

### 5.6 提醒兜底流程

```
任务进入临近到期 → 系统尝试微信提醒 → 首页持续展示临近到期/逾期状态 → 用户处理任务
```

微信提醒是优先提醒方式，不是唯一提醒方式。

## 7. 任务模型与状态

### 7.1 任务字段

| 字段 | 说明 |
|------|------|
| 所属家庭 | 任务归属的家庭空间 |
| 标题 | 任务名称 |
| 描述 | 可选说明 |
| 执行人 | 负责完成任务的家庭成员 |
| 创建人 | 创建任务的用户 |
| 截止时间 | 用于提醒和逾期判断 |
| 难度 | 1-5 星，默认可为空或 1 星 |
| 重复规则 | 不重复、每天、每周 |
| 是否需要验收 | 默认否 |
| 完成备注 | 可选 |
| 完成照片 | 可选，最多 1 张 |
| 驳回原因 | 驳回时必填 |
| 提醒状态 | 记录是否已尝试提醒 |

### 7.2 任务状态

| 状态 | 触发方式 | 说明 |
|------|----------|------|
| 待完成 | 创建并分配后进入 | 执行人需要处理 |
| 临近到期 | 系统计算状态 | 截止前 1 小时内，用于首页提醒 |
| 已逾期 | 系统计算状态 | 超过截止时间且未完成 |
| 待验收 | 成员标记完成，且任务需要验收 | 等待组织者确认 |
| 已完成 | 成员完成且无需验收，或组织者验收通过 | 终态 |
| 已驳回 | 组织者驳回 | 需要执行人重新处理 |
| 已取消 | 组织者取消任务 | 终态 |

临近到期和已逾期可作为计算状态，不一定需要独立存储。

## 10. 验收标准

### 10.1 家庭与成员

| AC-ID | 验收标准 |
|-------|----------|
| AC-01 | 用户可通过微信登录进入系统，首次登录后可创建或加入家庭 |
| AC-02 | 家庭创建成功后，系统自动生成邀请链接/二维码 |
| AC-03 | 被邀请者打开有效邀请链接后，可完成登录并加入对应家庭 |
| AC-04 | 一个用户可以加入多个家庭，并能切换当前家庭 |
| AC-05 | 页面和通知中必须明确展示任务所属家庭，避免多家庭混淆 |

### 10.2 任务创建

| AC-ID | 验收标准 |
|-------|----------|
| AC-06 | 用户可从内置家务模板创建任务，模板至少包含标题、默认难度、建议重复规则 |
| AC-07 | 用户可手动创建任务，必填项包括标题、执行人、截止时间 |
| AC-08 | 任务可设置重复规则：不重复、每天、每周 |
| AC-09 | 任务可设置是否需要验收，默认值为无需验收 |
| AC-10 | 创建任务流程在移动端 30 秒内可完成，不超过 3 个主要步骤 |

### 10.3 任务执行

| AC-ID | 验收标准 |
|-------|----------|
| AC-11 | 首页默认展示当前用户在当前家庭下的待完成、临近到期、逾期任务 |
| AC-12 | 用户可将自己的任务标记为完成，完成备注可选，照片可选 |
| AC-13 | 无需验收的任务在标记完成后直接进入已完成状态 |
| AC-14 | 需要验收的任务在标记完成后进入待验收状态 |
| AC-15 | 组织者可取消未完成、待验收或已驳回任务，取消后不再提醒 |

### 10.4 验收与驳回

| AC-ID | 验收标准 |
|-------|----------|
| AC-16 | 组织者可对待验收任务进行通过或驳回操作 |
| AC-17 | 驳回任务时必须填写驳回原因 |
| AC-18 | 驳回后任务进入已驳回状态，并允许执行人重新提交 |

### 10.5 重复任务

| AC-ID | 验收标准 |
|-------|----------|
| AC-19 | 每天重复任务完成后，系统生成下一天同一设置的新任务 |
| AC-20 | 每周重复任务完成后，系统生成下一周同一设置的新任务 |
| AC-21 | 重复生成的新任务继承原任务的标题、执行人、验收设置和提醒设置 |

### 10.6 提醒与兜底

| AC-ID | 验收标准 |
|-------|----------|
| AC-22 | 系统默认在任务到期前 1 小时尝试发送微信提醒 |
| AC-23 | 用户关闭提醒后，系统不再向该用户发送微信提醒 |
| AC-24 | 临近到期和逾期任务必须在首页突出展示，即使微信提醒未送达 |
| AC-25 | 通知内容必须包含任务名称、截止时间、所属家庭 |
| AC-26 | 系统记录提醒发送状态，用于避免同一任务重复提醒 |

### 10.7 轻量公平感

| AC-ID | 验收标准 |
|-------|----------|
| AC-27 | 家庭看板展示每个成员当前待办数 |
| AC-28 | 家庭看板展示每个成员本周完成数 |
| AC-29 | MVP 不展示排行榜、排名、惩罚标记或羞辱式提醒 |

## 12. 安全、隐私与数据边界

### 12.1 安全要求

| 类别 | 要求 |
|------|------|
| 认证 | 微信 OAuth 登录后由服务端签发 JWT |
| 鉴权 | 所有家庭、任务、成员接口必须校验用户是否属于对应家庭 |
| 角色权限 | 组织者可管理家庭和任务；成员只能操作自己有权限的任务 |
| 多家庭隔离 | 当前家庭上下文必须明确，禁止跨家庭读取或修改任务 |
| 邀请安全 | 邀请链接/二维码应包含有效期，过期后不能加入 |
| 上传安全 | 限制图片类型、大小和数量，文件名随机化，禁止直接暴露本地路径 |
| API 防护 | 登录、邀请、上传、提醒相关接口需要限流 |

### 12.2 隐私与数据边界

| 数据 | 边界 |
|------|------|
| 微信身份信息 | 只存储业务必要字段，例如 openid、昵称、头像；不存储无关微信资料 |
| 家庭关系 | 仅家庭成员可见，不对外公开 |
| 任务内容 | 仅所属家庭成员可见 |
| 完成照片 | 仅所属家庭成员可见，MVP 不做公开分享 |
| 孩子信息 | 不采集真实年龄、学校等敏感信息；"孩子"只作为家庭内角色 |
| 提醒设置 | 用户可关闭个人提醒，不影响任务本身 |
| 数据删除 | 用户可退出家庭；账号注销时应处理个人身份数据和任务历史归属 |

### 12.3 照片上传边界

- 照片上传是可选能力。
- 单次最多上传 1 张。
- 单张图片压缩后存储。
- 仅家庭成员可见。
- MVP 不做人脸识别、内容识别、公开分享。

## 13. 明确不做

- 不做原生 App。
- 不做 App 打包。
- 不做应用商店发布。
- 不做多租户商业 SaaS 能力。
- 不做复杂运营后台。
- 不做企业级项目管理能力，例如子任务、依赖、甘特图。
- 不做实时聊天。
- 不做公开社交。
- 不做排行榜。
- 不做惩罚机制。
- 不做公开批评或羞辱式提醒。
- 不做积分和兑换系统。
- 不做贡献统计图表。
- 不做购物清单。
- 不做共享日历。
- 不做智能自动分配。
- 不做复杂重复规则，例如隔周、每月、自定义 cron。
- 不做多语言。

## 14. 术语表

| 术语 | 定义 |
|------|------|
| 家庭 | 一个协作空间，成员在其中共享任务 |
| 家庭组织者 | 创建或管理家庭的人，拥有家庭和任务管理权限 |
| 家庭成员 | 加入家庭并参与任务协作的用户 |
| 孩子/青少年成员 | 家庭内成员角色，不代表系统采集儿童敏感身份信息 |
| 任务 | 一件需要完成的家务事项 |
| 任务模板 | 用于快速创建常见家务任务的预设内容 |
| 重复任务 | 按每天或每周自动生成下一次的周期性任务 |
| 验收开关 | 控制任务完成后是否需要组织者确认 |
| 验收 | 组织者确认任务完成的动作 |
| 驳回 | 组织者认为任务未达要求并要求重新处理的动作 |
| 临近到期 | 任务截止前 1 小时内的计算状态 |
| 已逾期 | 超过截止时间且未完成的计算状态 |
| 首页提醒兜底 | 当微信提醒不可用时，通过首页任务状态继续提醒用户 |
| "无唠叨" | 用系统提醒和清晰看板替代人际催促的核心设计理念 |
